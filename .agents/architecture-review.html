5 deepening opportunities found across agent, matching, pipeline, and UI modules

1. Collapse Twin Agent Factories

Strong
Duplicate Shallow
Files

src/lib/agent/graph.ts

Problem

createDecisionAgent and createCareAgent are ~90% identical — 105 lines of copy-paste that differ in only 4 lines: the system prompt string, the parameter name, and the tool arg key checked in toolNode. The care agent reuses state.planningProfileId to hold the owned profile ID — a semantic mismatch waiting to bite.

Before / After

Before — two factory functions

createDecisionAgent(profileId, repo, tools) // ~60 lines of LangGraph wiring createCareAgent(profileId, repo, tools) // ~60 lines of IDENTICAL wiring

After — one factory

createAgent({ profileId, repo, tools, systemPrompt }) // ~60 lines of LangGraph wiring once
Benefits
  • Locality: LangGraph wiring lives in one place. If the toolNode pattern changes, it's fixed once.
  • Leverage: Adding a third agent (e.g., Onboarding Agent) is 5 lines instead of 60.
  • Safety: Removes the semantic mismatch where care reuses planningProfileId state for owned profiles.

2. Centralize LLM Configuration

Strong
Duplicate
Files

src/lib/agent/graph.ts

src/app/api/match/follow-up/route.ts

Problem

Model name "deepseek/deepseek-chat", API key env var, and base URL are hardcoded in 3 places. The follow-up route uses raw fetch() while the agent uses LangChain's ChatOpenAI — same model, different invocation path. Changing models would require 3 edits that can drift.

Before — 3 hardcoded call sites

// graph.ts x2 new ChatOpenAI({ model: "deepseek/deepseek-chat", ... }) // follow-up/route.ts fetch("openrouter/chat/completions", { body: JSON.stringify({ model: "deepseek/deepseek-chat", ... }) })

After — single config source

import { getChatModel, llmConfig } from "@/lib/llm/config" // graph.ts const model = getChatModel({ temperature: 0.7 }) // follow-up/route.ts const result = await llmConfig.call({ messages, max_tokens: 500 })
Benefits
  • Locality: Model choice lives in one file. Swap DeepSeek → GPT-4o and every call site updates automatically.
  • Leverage: Adding retry logic, rate limiting, or cost tracking behind the LLM config module benefits all callers.
  • Testability: Follow-up route can now be tested with a mock LLM client instead of needing a real API key.

3. Fix DraftPlanningRepository's Supabase Leak

Strong
Leaky Interface
Files

src/lib/agent/draft-repo.ts

src/lib/agent/repository.ts

Problem

DraftPlanningRepository accepts a PlanningRepository via its constructor but never calls the delegate's write methods — it calls getSupabaseAdmin() directly to insert drafts. This means:

  • You cannot test draft behavior with a mock repo — the mock would be ignored for writes
  • The class name says "I wrap any PlanningRepository" but the implementation says "I write to Supabase regardless"
  • Care agent drafts reuse the planning_pet_profile_id column to store owned profile IDs — a semantic corruption waiting for a foreign key or join to expose it
  • current_value snapshotting is inconsistent: 4 of 6 write methods capture the before-state, 2 don't
This wrapper is a decorator that ignores its delegate for writes. It violates the substitution principle: swapping the real repo for an in-memory fake would leave drafts writing to real Supabase.

Before — concrete Supabase dependency leaked

class DraftPlanningRepository implements PlanningRepository { constructor(private real: PlanningRepository) {} async replaceExpenses(...) { const supabase = getSupabaseAdmin(); // ← skips this.real await supabase.from("agent_drafts").insert(...) } }

After — inject draftStore at the seam

class DraftPlanningRepository { constructor( private real: PlanningRepository, private drafts: DraftStore // ← injectable ) {} async replaceExpenses(...) { await this.drafts.create({ target: "estimated_expenses", proposed_value: expenses, current_value: await this.real.getExpenses(...) }) } }
Benefits
  • Testability: Inject an in-memory draft store; verify drafts are created without touching a database.
  • Locality: Draft storage logic (Supabase table vs. Redis vs. queue) is behind a seam. Change one adapter, not 6 methods.
  • Correctness: current_value snapshots become uniform because the wrapper delegates reads to this.real consistently.

4. Extract Shared Agent Route Pattern

Worth Exploring
Duplicate
Files

src/app/api/agent/chat/route.ts

src/app/api/agent/care/route.ts

Problem

Both agent routes share an 8-step pipeline: parse body → auth check → profile ownership verification → instantiate repo/tools/agent → invoke → extract response → return JSON. The care route is missing two features the chat route has: thread persistence and draft wrapping. This asymmetry is either a bug or undocumented.

Before — 72 + 46 lines, uneven features

chat/route.ts: ✅ Thread persistence ✅ Draft wrapping ✅ Returns thread_id care/route.ts: ❌ No thread persistence ❌ No draft wrapping ❌ No thread_id

After — shared helper, uniform features

// routes become 15 lines each export POST = runAgentRoute({ agentType: "decision", profileIdField: "planningProfileId", table: "planning_pet_profiles" })
Benefits
  • Locality: Auth + ownership + invocation lives in one helper. Bug fixes (e.g., better error handling) apply everywhere.
  • Leverage: Care agent gets thread persistence and draft support for free when the shared helper adds them.
  • Testability: The shared helper can be tested once with a mock agent; each route just tests its route-specific config.

5. Consolidate Tool Error-Handling Boilerplate

Worth Exploring
Duplicate
Files

src/lib/agent/tools.ts

src/lib/agent/care-tools.ts

Problem

Every single agent tool (8 total across both files) wraps its implementation in identical try/catch boilerplate:

async ({ ... }) => { try { await repo.someMethod(...); return "Success message"; } catch (e) { return `Error: ${e instanceof Error ? e.message : String(e)}`; } }

This is repeated 8 times verbatim. A safeTool(repoCallback) wrapper would collapse ~48 lines of noise into 8 lines of intent.

Benefits
  • Locality: Error formatting changes once. If you want to log errors, add retry logic, or change the error message format — one place.
  • Leverage: New tools are 5 lines instead of 13. The wrapper makes the tool's domain logic the only interesting part.

Top Recommendation

Start with #1 — Collapse Twin Agent Factories.

It's the highest-leverage change: 105 lines of copy-paste vanish, the semantic bug (planningProfileId reused for owned profiles) disappears, and adding a third agent becomes trivial. It's also the lowest-risk — no database changes, no API surface changes, pure internal refactor validated by the same 46 tests that already cover the matching and validation modules.

After #1, tackle #3 — Fix the DraftPlanningRepository leak. The draft module is the worst architectural smell: a decorator that ignores its delegate. Fixing it makes agent tests actually testable without a database.
